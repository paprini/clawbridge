# Configuration UX — openclaw-a2a

## The Problem

**Too many config files = User confusion + Security mistakes**

Current design has:
- `config/peers.json` (peer list)
- `config/a2a-skills.json` (skill whitelist)
- `.env` (tokens, URLs)
- Security group rules (AWS)
- systemd config

**Result:** Easy to misconfigure → accidental exposure OR nothing works

---

## Design Principle: Secure by Default, Simple to Advanced

**Tiers of configuration:**
1. **Zero config** — works out of the box (private, safe)
2. **Simple config** — one command to add peer
3. **Advanced config** — full control for power users

---

## Solution: Progressive Configuration

### Tier 1: Zero Config (Install → Works)

**User experience:**
```bash
# Install skill
clawhub install openclaw-a2a

# Run setup (automated)
cd ~/.openclaw/skills/openclaw-a2a
./setup.sh

# Answer 2 questions:
? What's this agent's name? (default: hostname)
> discord-bot

? Network mode? (default: private)
> private

✅ Done! Sidecar running on :9100 (VPC-only)
```

**What setup.sh does automatically:**
1. Generates bearer token (random, 32 chars)
2. Creates systemd service
3. Configures firewall (VPC-only)
4. Starts sidecar
5. Shows Agent Card URL

**Result:** Secure by default, no peers configured (isolated)

---

### Tier 2: Simple Peer Management (CLI)

**Add a peer (one command):**
```bash
openclaw-a2a peer add whatsapp \
  --url http://10.0.1.11:9100 \
  --token <their-token>

✅ Peer 'whatsapp' added
✅ Testing connection... SUCCESS
✅ Agent Card fetched: Guali WhatsApp (3 skills)
```

**What it does:**
1. Adds peer to `config/peers.json`
2. Tests connectivity
3. Fetches their Agent Card
4. Shows what skills they expose
5. No manual JSON editing

**List peers:**
```bash
openclaw-a2a peer list

Configured Peers:
├── whatsapp (http://10.0.1.11:9100) ✅ online
└── telegram (http://10.0.1.12:9100) ⏸️  disabled
```

**Remove peer:**
```bash
openclaw-a2a peer remove telegram

? Are you sure? (y/N)
> y

✅ Peer 'telegram' removed
```

### Tier 3: Skill Management (Safe Defaults)

**Default exposed skills (ultra-conservative):**

```bash
Exposed via A2A:
```


## Configuration File Design

### Single Config File (Not 3 Separate Files)

**File:** `~/.openclaw/skills/openclaw-a2a/config.yaml`

```yaml
# openclaw-a2a configuration
# Generated by setup.sh — edit carefully

agent:
  name: discord-bot
  version: 1.0.0
  network_mode: private  # private | public | hybrid

network:
  port: 9100
  bind: 127.0.0.1  # localhost for VPC-only

auth:
  bearer_token_file: ./.secrets/token.txt  # Auto-generated
  
peers:
  - id: whatsapp
    name: Guali WhatsApp
    url: http://10.0.1.11:9100
    token_file: ./.secrets/whatsapp-token.txt
    enabled: true
    
  - id: telegram
    name: Guali Telegram
    url: http://10.0.1.12:9100
    token_file: ./.secrets/telegram-token.txt
    enabled: false

skills:
  exposed:
    - ping
    - get_status
  
  available:
    - name: search
      public: false
      reason: "Can leak private conversations"
    - name: send_message
      public: false
      reason: "Can spam or impersonate"
```

**Benefits:**
- One file to understand
- Comments explain each setting
- Tokens in separate files (not in config)
- Clear public/private distinction

## Phase 2: Public Discovery (Simple Mode)

**User wants to make agent public:**

```bash
openclaw-a2a config set network_mode public

⚠️  WARNING: This will make your agent discoverable by anyone
⚠️  Only exposed skills will be available to external agents

Current exposed skills:
├── ping
└── get_status

? Proceed? (y/N)
> y

What changes:
├── Port 9100 opened to internet (security group updated)
├── Agent Card published to A2A registry
└── Public URL: https://discord-bot.a2a.openclaw.ai

✅ Agent is now publicly discoverable
ℹ️  To expose more skills: openclaw-a2a skills expose <skill>
```

**What setup does:**
1. Updates AWS security group (automates firewall change)
2. Generates public Agent Card (limited skills)
3. Registers with A2A directory
4. Sets up HTTPS (Let's Encrypt)
5. Keeps private skills private

## Advanced: Hybrid Mode (Best of Both)

**User wants both private peers + public discovery:**

```bash
openclaw-a2a config set network_mode hybrid

Hybrid mode configuration:
├── Private peers: whatsapp, telegram (full access)
└── Public access: limited skills only

? Proceed? (y/N)
> y

✅ Hybrid mode enabled
```

**How it works:**
- Private Agent Card: `http://10.0.1.10:9100/.well-known/agent-card.json` (VPC-only, full skills)
- Public Agent Card: `https://discord-bot.a2a.openclaw.ai/.well-known/agent-card.json` (internet, limited skills)

**Config:**
```yaml
network_mode: hybrid

skills:
  private_peers:  # Full access for trusted peers
    - ping
    - get_status
    - search
    - send_message
  
  public:  # Limited for external agents
    - ping
    - get_status
```

## Configuration Templates (Smart Defaults)

### Template 1: Family Network (Most Common)

```bash
openclaw-a2a init --template family

This template sets up:
├── Private network (VPC-only)
├── 2-3 peer slots (for your other instances)
├── Conservative skill exposure (ping, status only)
└── Shared bearer token

Perfect for: Multiple OpenClaw instances you control

Continue? (Y/n)
```

### Template 2: Public Agent

```bash
openclaw-a2a init --template public

This template sets up:
├── Public discovery (A2A registry)
├── HTTPS with Let's Encrypt
├── Minimal skill exposure (metadata only)
├── Rate limiting (100 req/min)
└── Public bearer tokens

Perfect for: Community-facing agent (e.g., Musicate QA bot)

Continue? (Y/n)
```

### Template 3: Development

```bash
openclaw-a2a init --template dev

This template sets up:
├── Localhost-only (127.0.0.1)
├── No auth required (dev mode)
├── All skills exposed (for testing)
└── Verbose logging

⚠️  DO NOT use in production

Perfect for: Testing A2A integration locally

Continue? (Y/n)
```

## Error Prevention

### Common Mistakes Prevented:

**Mistake 1: Exposing search without scope**
```bash
openclaw-a2a skills expose search

❌ ERROR: Skill 'search' is too broad

This skill can access all channels and files.

Suggestions:
├── Create scoped variant: search_public_docs
├── Or confirm you understand the risk: --force
```

**Mistake 2: Public mode with private skills exposed**
```bash
openclaw-a2a config set network_mode public

❌ ERROR: Cannot enable public mode

You have private skills exposed:
├── search (can leak conversations)
└── send_message (can spam)

Fix:
$ openclaw-a2a skills hide search
$ openclaw-a2a skills hide send_message
```

**Mistake 3: Firewall misconfiguration**
```bash
openclaw-a2a peer add telegram --url http://10.0.1.12:9100

Testing connection...
❌ Connection failed: timeout

Troubleshooting:
├── Is peer agent running? (check systemctl status openclaw-a2a)
├── Firewall blocking? (check security group allows :9100 from this IP)
├── Wrong URL? (verify IP address)

Run diagnostics: openclaw-a2a diagnose telegram
```

```markdown
- Test URL manually: curl <url>/health
```
